# v0.8.2 Plan: Multimodal I/O Modules (Vision + TTS)

Last updated: 2026-03-05  
Status: Planned / foundation in place

## Objective

Add optional capability modules for:

1. Screen-context input (`input_vision`)
2. Voice output (`output_voice_tts`)

and bridge them into workflow-agent execution without degrading core transcription reliability.

## Product Defaults

1. Vision: low-fps stream.
2. Vision scope: all monitors (policy default).
3. Retention: RAM-only, no frame disk persistence.
4. TTS output policy: agent replies only.
5. TTS provider strategy: dual provider lanes (`windows_native`, `local_custom`) with fallback.
6. Platform: Windows-first rollout.

## API Surface

### Vision Commands

1. `list_screen_sources`
2. `start_vision_stream`
3. `stop_vision_stream`
4. `get_vision_stream_health`
5. `capture_vision_snapshot`

### TTS Commands

1. `list_tts_providers`
2. `list_tts_voices`
3. `speak_tts`
4. `stop_tts`
5. `test_tts_provider`

### Events

1. `vision:stream-started`
2. `vision:stream-stopped`
3. `vision:stream-error`
4. `vision:frame-meta`
5. `tts:speech-started`
6. `tts:speech-finished`
7. `tts:speech-error`

### Settings

1. `vision_input_settings`
2. `voice_output_settings`

## Milestone Slices

1. N1/N2: schema + module registry + permissions.
2. N3/N4: command/event surface (foundation delivered).
3. N5/N6: runtime hardening for real frame pipeline and custom local TTS backend.
4. N7/N8: workflow-agent capability bridge + output policy enforcement.
5. N9-N14: privacy hardening, benchmarks, E2E, release packet.

## Verification

1. Vision stream health transitions are correct.
2. No disk writes for frame payloads.
3. TTS start/finish/error events fire consistently.
4. Provider fallback path is deterministic.
5. Benchmark suite runs >=3 passes/provider/scenario before default-provider decision.
